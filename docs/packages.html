Packages & Modules

Snafu's module system lets you organize code into reusable .snf files, control what names are visible externally, and seamlessly bridge to Python's ecosystem.

Module = File

Each .snf file is a module. All names defined at the top level of the file become module-level bindings. By default, all top-level names (except those starting with __) are exported.

greetings.snf
# A simple module with two functions

df hello(name) = "Hello, " + name + "!"
df goodbye(name) = "Goodbye, " + name + "."

Importing with us

The us (use) keyword imports a module. It loads the .snf file from the current working directory, executes it in a fresh interpreter, caches the result, and makes the exported names available.

Qualified import

Qualified: access via module.name
us greetings

p(greetings.hello("Alice"))       # Hello, Alice!
p(greetings.goodbye("Bob"))       # Goodbye, Bob.
Hello, Alice!
Goodbye, Bob.

Wildcard import

Wildcard: dump all names into scope
us *greetings

p(hello("Alice"))                # Hello, Alice!
Hello, Alice!

Aliased import

Alias: rename the module namespace
us greetings as gr

p(gr.hello("Alice"))              # Hello, Alice!
Hello, Alice!

Caching: Modules are loaded once. Subsequent us of the same module returns the cached scope. The child interpreter inherits the parent's strict-mode setting.

Controlling Exports with xp

By default, all top-level names are exported. Use xp (export) to restrict which names are visible to importers.

math_helpers.snf -- only add and mul are exported
xp [add, mul]

df add(a, b) = a + b
df mul(a, b) = a * b
df __internal_helper(x) = x * x   # not exported (not in xp list)
Using the module
us math_helpers

p(math_helpers.add(3, 4))        # 7
p(math_helpers.mul(3, 4))        # 12
# math_helpers.__internal_helper  -- not accessible
7
12

How it works: xp stores the names list as __exports__ in the module's scope. When another module imports via us, the importer checks for __exports__ and only exposes those names.

Python Interop with py.import()

Snafu can directly import and use any Python module via py.import(name). This calls Python's importlib.import_module under the hood.

Importing Python modules explicitly
math = py.import("math")
p(math.sqrt(16))                   # 4.0
p(math.pi)                         # 3.141592653589793

json = py.import("json")
p(json.dumps([1, 2, 3]))          # [1, 2, 3]
4.0
3.141592653589793
[1, 2, 3]

Value interop: Primitives, lists, dicts, and tuples round-trip by conversion. Python callables are wrapped with automatic argument/return conversion. NumPy arrays and pandas DataFrames are opaque handles with method access.

Limits:

  • Symbolic numbers must be flattened with .fl() before entering NumPy.
  • Algebraic effects do not cross into Python and back.
  • Python metaclasses are not respected by Snafu dispatch.

Name Resolution Order

When Snafu encounters a name, it searches in this order:

  1. Local scope
  2. Enclosing lexical scopes
  3. Module-level scope
  4. Active us blocks (last wins on collision)
  5. Snafu built-ins (prelude)
  6. Python alias table (triggers import if not yet loaded)
  7. und (or NameErr in strict mode)

Creating a Reusable Module

Here is a complete walkthrough of creating and using a math utilities module.

Step 1: Create the module file

lib/math_utils.snf
# Math utilities module
xp [clamp, lerp, smoothstep, deg2rad, rad2deg, dist, normalize]

df clamp(x, lo, hi) = min([max([x, lo]), hi])
df lerp(a, b, t) = a + (b - a) * t
df smoothstep(edge0, edge1, x) {
    t = clamp((x - edge0) / (edge1 - edge0), 0, 1)
    t * t * (3 - 2 * t)
}
df deg2rad(d) = d * pi / 180
df rad2deg(r) = r * 180 / pi
df dist(x1, y1, x2, y2) = sqrt((x2-x1)**2 + (y2-y1)**2)
df normalize(lst) {
    total = sum(lst)
    if total == 0 { lst }
    el { m(lst, `a / total`) }
}

Step 2: Use it from another file

my_program.snf
us math_utils

# Clamp a value to a range
p(math_utils.clamp(15, 0, 10))    # 10

# Linear interpolation
p(math_utils.lerp(0, 100, 0.5))  # 50

# Distance between two points
p(math_utils.dist(0, 0, 3, 4))   # 5.0

# Convert degrees to radians
p(math_utils.deg2rad(180))       # 3.14159...

# Normalize a probability distribution
p(math_utils.normalize([2, 3, 5]))  # [0.2, 0.3, 0.5]
10
50
5.0
3.14159...
[0.2, 0.3, 0.5]

Step 3: Wildcard import for brevity

Shorter usage with us *
us *math_utils

# Now use functions without prefix
p(lerp(0, 100, 0.75))             # 75
p(dist(1, 1, 4, 5))               # 5.0
75
5.0

Example: String Utilities Module

lib/string_utils.snf
xp [capitalize, title_case, pad_center, truncate, slug]

df capitalize(s) = s[0].upr() + s[1:]
df title_case(s) = J(" ", m(S(s, " "), `capitalize(a)`))
df pad_center(s, width, ch=" ") = s.cpad(width, ch)
df truncate(s, n, suffix="...") = if len(s) <= n { s } el { s[:n-len(suffix)] + suffix }
df slug(s) = J("-", m(S(s.lwr(), " "), `a`))
Using string_utils
us string_utils as su

p(su.capitalize("hello"))         # Hello
p(su.title_case("hello world"))    # Hello World
p(su.truncate("a long string", 8)) # a lon...
p(su.slug("Hello World Test"))     # hello-world-test
Hello
Hello World
a lon...
hello-world-test

Module Best Practices

  • Always declare xp [...] to make your public API explicit.
  • Prefix internal helpers with __ as a convention (they are hidden by default even without xp).
  • Place reusable modules in a lib/ directory.
  • Modules are cached after first load -- side effects in the module body run only once.
  • Use us module as alias to avoid name collisions when importing multiple modules.
  • Use us *module sparingly -- it can pollute the namespace. Prefer qualified imports for larger projects.

Import Cycles

Circular imports are detected at resolution time. By default, Snafu tolerates them (the partially-initialized module is used). For stricter checking, add this at the top of your program:

Strict cycle detection
opt strict_cycles true
